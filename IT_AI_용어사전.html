Next.js / App Router
넥스트제이에스 / 앱 라우터
🛠 프레임워크
React 기반의 풀스택 웹 프레임워크. App Router는 Next.js 13+에서 도입된 라우팅 방식으로, 파일 구조가 곧 URL 경로가 된다. 서버 컴포넌트, 서버 액션, 스트리밍을 기본 지원한다.
예시
src/app/teams/[teamId]/page.tsx 파일을 만들면 /teams/abc123 URL이 자동 생성된다.
Server / Client Component
서버 컴포넌트 / 클라이언트 컴포넌트
🛠 프레임워크
서버 컴포넌트는 서버에서만 실행되는 React 컴포넌트. DB 접근, 환경변수 사용 가능. 클라이언트 컴포넌트는 브라우저에서도 실행되며 파일 상단에 'use client'를 선언한다. 버튼 클릭, 폼 입력 등 인터랙션이 필요할 때만 사용한다.
예시
태스크 목록 조회는 서버 컴포넌트, 태스크 생성 폼은 'use client' 클라이언트 컴포넌트로 분리.
Server Action
서버 액션
🛠 프레임워크
Next.js에서 클라이언트 폼 제출을 별도 API 엔드포인트 없이 서버 함수로 직접 처리하는 방식. 'use server'를 선언한 함수가 자동으로 POST 엔드포인트가 된다. DB 접근, 인증 확인, 입력 검증을 한 곳에서 처리한다.
예시
createTask(formData) 함수에 'use server' 선언 → 폼 제출 시 서버에서 실행, DB에 태스크 저장.
TypeScript (strict mode)
타입스크립트 엄격 모드
🛠 프레임워크
JavaScript에 정적 타입을 추가한 언어. strict 모드에서는 any 타입 사용이 사실상 금지되고, null·undefined 처리를 강제해 런타임 오류를 사전에 잡는다.
예시
const title: string = formData.get('title') as string — 타입을 명시하지 않으면 컴파일 오류 발생.
Supabase
수파베이스
🛠 프레임워크
PostgreSQL 기반의 오픈소스 백엔드 플랫폼. Auth(인증), DB, 스토리지, Edge Functions, Realtime을 하나의 서비스로 제공한다. Firebase의 오픈소스 대안으로 불린다.
예시
Supabase 프로젝트 생성 → DB + Auth 즉시 사용 가능. RLS 정책으로 팀별 데이터 격리.
Vercel
버셀
🛠 프레임워크
Next.js를 만든 회사가 운영하는 프론트엔드 배포 플랫폼. GitHub 저장소를 연결하면 push할 때마다 자동으로 빌드·배포된다. Preview URL로 PR마다 독립적인 미리보기 환경을 제공한다.
예시
git push origin main → Vercel이 자동 감지 → taskflow.vercel.app에 배포 완료.
pnpm
피엔피엠
🛠 프레임워크
Node.js 패키지 관리자. npm보다 디스크 공간을 적게 쓰고 설치 속도가 빠르다. 패키지를 전역 저장소에 한 번만 저장하고 심볼릭 링크로 연결하는 방식을 사용한다.
예시
pnpm install, pnpm dev, pnpm build — TaskFlow 프로젝트 기본 명령어.
RLS (Row Level Security)
행 수준 보안
🗄 데이터베이스
PostgreSQL의 기능으로, 테이블 행(row) 단위로 접근 권한을 제어한다. 같은 테이블이라도 사용자에 따라 볼 수 있는 데이터가 달라진다. ENABLE ROW LEVEL SECURITY 선언 후 정책(POLICY)을 추가해 사용한다.
예시
팀A 멤버는 팀A 태스크만 조회. 팀B 멤버는 같은 tasks 테이블을 봐도 팀B 데이터만 반환됨.
ERD (Entity Relationship Diagram)
엔티티 관계 다이어그램
🗄 데이터베이스
데이터베이스의 테이블(엔티티)과 그 관계를 시각적으로 표현한 다이어그램. 어떤 테이블이 존재하고, 어떤 컬럼을 가지며, 서로 어떻게 연결되는지를 한눈에 파악할 수 있다.
예시
teams 1 → N members / members N → 1 tasks — TaskFlow ERD의 핵심 관계.
Soft Delete
소프트 삭제
🗄 데이터베이스
데이터를 실제로 삭제하지 않고, deleted_at 컬럼에 삭제 시각을 기록하는 패턴. 데이터 복구·감사(audit)·RLS SELECT 정책 적용이 용이해진다.
예시
삭제 버튼 클릭 → UPDATE tasks SET deleted_at = NOW(). 목록 조회 시 WHERE deleted_at IS NULL으로 필터.
PostgreSQL
포스트그레에스큐엘
🗄 데이터베이스
오픈소스 관계형 데이터베이스. RLS, 트리거, UUID, JSON 타입, pgvector 확장 등 고급 기능을 지원한다. Supabase의 핵심 DB 엔진이다.
예시
gen_random_uuid()로 UUID 자동 생성, TIMESTAMPTZ로 시간대 포함 시각 저장.
DB Trigger
데이터베이스 트리거
🗄 데이터베이스
INSERT / UPDATE / DELETE 같은 DB 이벤트 발생 시 자동으로 실행되는 함수. 앱 코드 없이 DB 레벨에서 데이터를 자동 처리한다.
예시
teams에 새 팀이 INSERT되면 → 트리거가 자동으로 생성자를 members에 leader로 추가.
Edge Function
엣지 함수
🗄 데이터베이스
Supabase/Vercel 등에서 제공하는 서버리스 함수. 사용자와 가까운 엣지 서버에서 실행돼 응답이 빠르다. 주로 외부 API 호출(이메일 발송, AI API 등)에 사용한다.
예시
멤버 초대 시 Supabase Edge Function → Resend API 호출 → 초대 이메일 발송.
pg_cron
피지크론
🗄 데이터베이스
PostgreSQL 내에서 SQL 작업을 주기적으로 자동 실행하는 확장(extension). Linux의 cron과 동일한 문법(* * * * *)을 사용한다.
예시
매일 자정 RLS가 꺼진 테이블을 감지해 audit_log에 기록 — 0 0 * * * 스케줄.
pgvector
피지벡터
🗄 데이터베이스
PostgreSQL에서 AI 임베딩 벡터를 저장하고 유사도 검색을 할 수 있게 해주는 확장. RAG(검색 증강 생성) 구현에 핵심적으로 사용된다.
예시
회의록 텍스트를 벡터로 변환 → pgvector에 저장 → "지난달 결정사항 알려줘" 질문 시 의미 기반 검색.
maybeSingle() vs single()
메이비싱글 vs 싱글
🗄 데이터베이스
Supabase JS SDK의 조회 메서드. single()은 결과가 정확히 1건이어야 하며, 0건이면 오류를 던진다. maybeSingle()은 0건이면 null을 반환해 안전하다.
예시
존재 여부가 불확실한 태스크 조회 → .maybeSingle() 사용. 반드시 있어야 하는 세션 → .single().
getUser() vs getSession()
유저 가져오기 vs 세션 가져오기
🔒 보안/인증
Supabase Auth의 두 가지 인증 확인 방법. getSession()은 JWT 서명을 서버에서 재검증하지 않아 위변조 위험이 있다. getUser()는 Supabase 서버에 요청해 JWT를 실제로 검증하므로 안전하다.
예시
모든 Server Action 최상단에 const { data: { user } } = await supabase.auth.getUser() 필수.
auth.uid() 패턴
현재 사용자 ID 참조
🔒 보안/인증
Supabase RLS 정책에서 현재 로그인한 사용자의 UUID를 가져오는 함수. auth.uid()를 직접 쓰면 행마다 N번 호출되어 느리다. (SELECT auth.uid())로 감싸면 쿼리당 1번만 호출되어 성능이 향상된다.
예시
❌ WHERE user_id = auth.uid() → ✅ WHERE user_id = (SELECT auth.uid())
service_role vs anon key
서비스 롤 키 vs 익명 키
🔒 보안/인증
service_role 키는 RLS를 우회하는 관리자 키로, 절대 클라이언트에 노출되면 안 된다. anon(익명) 키는 공개 키로 클라이언트에서 사용하며, RLS 정책이 적용된다.
예시
NEXT_PUBLIC_SUPABASE_ANON_KEY → 브라우저 허용. SUPABASE_SERVICE_ROLE_KEY → 서버 전용, .env.local에만 보관.
.env.local / 환경변수
환경 변수 파일
🔒 보안/인증
API 키, DB URL 같은 비밀값을 코드에 하드코딩하지 않고 환경 변수 파일에 분리하는 방식. .env.local은 반드시 .gitignore에 추가해 Git에 올라가지 않도록 해야 한다.
예시
NEXT_PUBLIC_SUPABASE_URL=https://xxx.supabase.co — NEXT_PUBLIC_ 접두사가 있으면 브라우저에 노출됨.
USING vs WITH CHECK
RLS 정책 조건 절
🔒 보안/인증
RLS 정책의 두 가지 조건 절. USING은 읽을 때 적용 (SELECT, UPDATE 대상 행 필터). WITH CHECK는 쓸 때 적용 (INSERT, UPDATE 후 결과 행 검증).
예시
SELECT 정책: USING (team_id IN (...)) / INSERT 정책: WITH CHECK (created_by = (SELECT auth.uid()))
SECURITY DEFINER
보안 정의자
🔒 보안/인증
PostgreSQL 함수 선언 옵션. 함수를 호출한 사용자의 권한이 아니라 함수를 정의한 소유자의 권한으로 실행된다. 트리거 함수에서 auth.uid()처럼 높은 권한이 필요할 때 사용한다.
예시
팀 생성 트리거에 SECURITY DEFINER 선언 → 일반 사용자가 팀을 만들어도 members 테이블에 삽입 가능.
JWT (JSON Web Token)
제이더블유티
🔒 보안/인증
사용자 인증 정보를 담은 서명된 토큰. 서버가 DB를 조회하지 않아도 토큰 서명만 검증해 사용자를 식별할 수 있다. Supabase Auth가 로그인 시 발급한다.
예시
로그인 → Supabase가 JWT 발급 → 쿠키에 저장 → 이후 요청마다 JWT를 서버에서 검증.
RAG (Retrieval-Augmented Generation)
검색 증강 생성
🤖 AI/LLM
AI가 답변을 생성할 때 외부 데이터베이스에서 관련 문서를 먼저 검색(Retrieval)하고, 그 내용을 참고해 답변을 생성(Generation)하는 방식. 모델이 학습하지 않은 최신 정보나 사내 문서를 답변에 활용할 수 있다.
예시
"지난 회의에서 결정한 내용이 뭐야?" → 회의록 DB에서 관련 문서 검색 → AI가 검색 결과를 바탕으로 답변 생성.
임베딩 (Embedding)
텍스트 → 벡터 변환
🤖 AI/LLM
텍스트, 이미지 같은 데이터를 의미(semantic)를 보존하는 숫자 배열(벡터)로 변환하는 기술. 의미가 비슷한 문장은 벡터 공간에서 가까운 거리에 위치한다. RAG의 핵심 기반 기술이다.
예시
"강아지"와 "개"는 임베딩 벡터가 가깝게 위치 → 유사 문서 검색에 활용.
MCP (Model Context Protocol)
모델 컨텍스트 프로토콜
🤖 AI/LLM
Claude 같은 AI 모델이 외부 도구(DB, GitHub, Notion 등)에 표준화된 방식으로 접근할 수 있도록 Anthropic이 설계한 프로토콜. 플러그인처럼 MCP 서버를 연결하면 Claude가 해당 시스템을 직접 조회·실행할 수 있다.
예시
Supabase MCP 연결 → Claude Code에서 "tasks 테이블 컬럼 뭐야?"라고 물으면 DB를 직접 조회해 답변.
Skill (Claude Code)
클로드 코드 스킬
🤖 AI/LLM
Claude Code에서 반복되는 작업 패턴을 SKILL.md 파일로 정의해 재사용하는 기능. 한 번 정의하면 같은 지시를 반복하지 않아도 일관된 결과를 얻을 수 있다.
예시
.claude/skills/rls-policy-review/SKILL.md 작성 → "RLS 검토해줘"라고 하면 정의된 4가지 패턴 자동 점검.
Claude / Claude API
클로드 / 클로드 API
🤖 AI/LLM
Anthropic이 만든 AI 언어 모델(LLM)과 그 API. Claude Code는 개발자용 CLI 도구로, 코드 작성·리뷰·실행을 AI와 함께 대화하며 처리한다. TaskFlow의 AI 우선순위 추천 기능에 Claude API를 사용한다.
예시
태스크 목록을 Claude API에 전달 → "오늘 먼저 처리해야 할 태스크 3개"를 AI가 분석해 반환.
Vertical Slice
수직 단면 개발
🏗 아키텍처
DB → 서버 로직 → UI까지 하나의 기능을 처음부터 끝까지 완성하는 개발 방식. 레이어(DB 전체 → API 전체 → UI 전체) 순서로 개발하는 방식과 대비된다.
예시
Slice 1: 회원가입 DB + 인증 API + 로그인 UI까지 완성 → Slice 2: 팀 생성 전체 완성.
Feature-based 폴더 구조
기능 중심 폴더 구조
🏗 아키텍처
파일을 종류(hooks/, services/, components/)가 아닌 도메인(features/tasks/, features/teams/) 기준으로 묶는 방식. 관련 파일이 한 폴더에 모여 유지보수가 쉽다.
예시
❌ /hooks/useTask.ts + /services/taskService.ts → ✅ /features/tasks/actions.ts + TaskList.tsx
SaaS (Software as a Service)
서비스형 소프트웨어
🏗 아키텍처
소프트웨어를 설치하지 않고 인터넷 브라우저로 구독해 사용하는 모델. 공급자가 서버·업데이트·보안을 관리한다. TaskFlow는 B2B SaaS — 팀 단위로 계정을 구매한다.
예시
Gmail, Slack, Notion, Linear, Asana — 모두 SaaS. 설치 없이 브라우저로 이용, 월정액 결제.
Multi-tenant (멀티 테넌트)
다중 고객 격리 구조
🏗 아키텍처
하나의 서버·DB에 여러 고객(팀)의 데이터가 공존하지만, 각 고객은 자신의 데이터만 볼 수 있도록 격리하는 아키텍처. TaskFlow에서는 team_id가 격리 기준이다.
예시
팀A와 팀B가 같은 tasks 테이블을 사용하지만, RLS가 team_id로 데이터를 격리.
ADR (Architecture Decision Record)
아키텍처 결정 기록
🏗 아키텍처
기술적 의사결정(어떤 기술을 왜 선택했는지)을 문서로 남기는 관행. 나중에 팀원이 "왜 이렇게 했지?"라고 물을 때 답을 제공한다. 선택·대안·이유·리스크·재검토 시점을 기록한다.
예시
ADR-001: "Next.js App Router 선택 — Server Actions로 API 없이 폼 처리 가능. 리스크: 캐싱 복잡도."
TRD (Technical Reference Document)
기술 참조 문서
🏗 아키텍처
기술 스택 선택, 아키텍처, API/Action 명세, 보안 정책, 성능 목표를 한 문서에 정리한 기술 설계서. PRD(무엇을 만들지)를 받아 개발팀이 어떻게 만들지를 결정한 결과물이다.
예시
TRD에 "Server Action 응답 < 200ms (p95), Lighthouse SEO 90+"를 명시해 성능 기준을 팀이 공유.
revalidatePath()
경로 캐시 무효화
🏗 아키텍처
Next.js에서 특정 경로의 캐시를 즉시 무효화하는 함수. Server Action에서 DB를 수정한 후 호출하면, 다음 요청 때 최신 데이터를 가져와 페이지가 갱신된다.
예시
태스크 생성 후 revalidatePath('/teams/abc/tasks') 호출 → 태스크 목록 페이지가 자동으로 새 데이터 표시.
GEO (Generative Engine Optimization)
생성형 엔진 최적화
📡 SEO/GEO
ChatGPT, Perplexity, Claude 같은 AI 검색 엔진이 제품을 정확히 인용·추천하도록 최적화하는 전략. 기존 SEO(Google 크롤러 최적화)와 함께 적용한다. 직접 답변 가능한 문장, 구체적 수치, 구조화 데이터(JSON-LD)가 핵심이다.
예시
"팀 태스크 관리 앱 추천해줘" → Perplexity가 TaskFlow를 인용하려면 GEO 전략이 필요.
SEO (Search Engine Optimization)
검색 엔진 최적화
📡 SEO/GEO
Google 같은 전통 검색 엔진에서 웹사이트가 상위에 노출되도록 최적화하는 작업. 메타태그, 사이트맵, robots.txt, 페이지 속도, 구조화 데이터 등을 관리한다.
예시
Lighthouse SEO 90점 이상, sitemap.xml 등록, 인증 후 페이지 noindex 처리.
JSON-LD
제이슨 엘디
📡 SEO/GEO
웹 페이지의 내용을 검색 엔진과 AI가 이해할 수 있도록 구조화된 데이터를 JSON 형식으로 HTML에 삽입하는 방식. Schema.org 표준을 따른다.
예시
SoftwareApplication 스키마로 "무료 팀 태스크 관리 SaaS"를 AI가 인식 → AI 검색 인용 확률 상승.
Open Graph (OG)
오픈그래프
📡 SEO/GEO
카카오톡, 슬랙, 트위터 등에서 링크를 공유할 때 나타나는 미리보기 카드를 제어하는 메타태그 표준. 제목, 설명, 이미지를 <meta property="og:*">로 지정한다.
예시
<meta property="og:title" content="TaskFlow — AI 태스크 관리"> → 카카오톡 공유 시 멋진 카드 노출.
Core Web Vitals (LCP · INP · CLS)
핵심 웹 성능 지표
📡 SEO/GEO
Google이 정의한 3가지 웹 성능 지표. LCP(Largest Contentful Paint): 주요 콘텐츠 로딩 시간(목표 <2.5s). INP(Interaction to Next Paint): 클릭 후 화면 반응 속도. CLS(Cumulative Layout Shift): 페이지 레이아웃 흔들림 방지.
예시
이미지 없이 텍스트만 먼저 로드 → 이미지 로드 후 레이아웃이 밀리면 CLS 점수 하락.
sitemap.xml / robots.txt
사이트맵 / 로봇 규칙
📡 SEO/GEO
sitemap.xml: 검색 엔진 크롤러에게 "이 URL들을 수집해"라고 알려주는 파일. robots.txt: 크롤러가 접근하면 안 되는 경로를 Disallow로 지정한다. 인증 후 개인 데이터 페이지는 반드시 Disallow 처리해야 한다.
예시
Disallow: /dashboard, Disallow: /teams/ → AI 크롤러가 팀 데이터 수집 불가.
PRD (Product Requirements Document)
제품 요구사항 문서
📊 제품/비즈니스
제품이 무엇을 해야 하는지를 정의하는 기획 문서. 목표, 타깃 사용자, 기능 목록(MoSCoW), 성공 지표, 리스크를 포함한다. 개발 전 이해관계자 간 합의의 기준이 된다.
예시
PRD Must: "Supabase Auth 로그인", "태스크 CRUD", "AI 우선순위 추천" — 없으면 제품이 동작하지 않는 기능들.
MoSCoW 우선순위
기능 우선순위 프레임워크
📊 제품/비즈니스
기능을 4단계로 분류하는 우선순위 프레임워크. Must: 반드시 있어야 함 / Should: 있으면 좋음 / Could: 여유 있으면 추가 / Won't: 이번 버전에서 제외.
예시
Must: 로그인·태스크 CRUD. Should: AI 추천. Could: 다크모드. Won't: 모바일 앱 (v1에서 제외).
North Star Metric
북극성 지표
📊 제품/비즈니스
제품의 핵심 가치를 가장 잘 나타내는 단 하나의 지표. 팀 전체가 이 지표를 기준으로 의사결정한다. 모든 기능이 이 지표에 기여해야 한다.
예시
TaskFlow의 North Star Metric: "월간 복귀 사용자 수" — 가입 후 떠난 팀이 돌아오는 비율.
TAM / SAM / SOM
시장 규모 분석 지표
📊 제품/비즈니스
TAM(Total Addressable Market): 이론적 전체 시장 규모. SAM(Serviceable Addressable Market): 실제로 공략 가능한 시장. SOM(Serviceable Obtainable Market): 현실적으로 점유 가능한 시장 규모.
예시
TAM: 글로벌 프로젝트 관리 SW 시장($7B) → SAM: 한국 스타트업 팀 → SOM: 1년 내 100개 팀 확보.
MVP (Minimum Viable Product)
최소 기능 제품
📊 제품/비즈니스
핵심 가치를 검증하기 위한 최소한의 기능만 갖춘 제품. 시장 반응을 빠르게 확인하고 리스크를 줄이기 위해 완성도보다 속도를 우선한다.
예시
TaskFlow MVP: 로그인 + 팀 생성 + 태스크 CRUD. AI 추천·다크모드는 MVP 이후 추가.
ARR (Annual Recurring Revenue)
연간 반복 매출
📊 제품/비즈니스
구독형 SaaS 비즈니스에서 1년 동안 반복해서 발생하는 예상 매출. 성장률을 측정하는 핵심 지표다. 월간 반복 매출(MRR) × 12로 계산한다.
예시
월 10만원 팀 100개 구독 → MRR 1,000만원 → ARR 1억 2,000만원.
INVEST (백로그 기준)
좋은 백로그 아이템의 조건
⚙️ 개발 방법론
좋은 백로그 아이템(User Story)의 6가지 조건 약자. Independent(독립적) · Negotiable(협의 가능) · Valuable(가치 있음) · Estimable(추정 가능) · Small(작음) · Testable(테스트 가능).
예시
"사용자는 태스크를 생성할 수 있다" — V(팀에 가치), T(태스크 목록에 즉시 반영되면 통과), E(2시간 추정).
CLAUDE.md
클로드 프로젝트 규칙 파일
⚙️ 개발 방법론
Claude Code에게 프로젝트 규칙을 알려주는 설정 파일. 기술 스택, 폴더 구조, 금지 사항, 검증 명령어를 명시하면 모든 대화에서 자동으로 규칙이 적용된다.
예시
"any 타입 금지", "pnpm build · lint · typecheck 통과 필수", "서버 컴포넌트 우선" 등을 CLAUDE.md에 명시.
CI/CD
지속적 통합 / 지속적 배포
⚙️ 개발 방법론
CI(Continuous Integration): 코드 푸시 시 자동으로 빌드·테스트를 실행해 오류를 즉시 발견. CD(Continuous Deployment): 테스트 통과 후 자동으로 프로덕션에 배포.
예시
git push → GitHub Actions가 pnpm build + lint 자동 실행 → 통과 시 Vercel 자동 배포.
CRUD
생성·조회·수정·삭제
⚙️ 개발 방법론
데이터의 4가지 기본 연산. Create(생성) · Read(조회) · Update(수정) · Delete(삭제). 대부분의 웹 애플리케이션 기능이 이 4가지 조합으로 이루어진다.
예시
태스크 생성(C) → 목록 조회(R) → 상태 수정(U) → 소프트 삭제(D).
lint / typecheck / build
코드 검증 삼총사
⚙️ 개발 방법론
lint: 코드 스타일·잠재적 버그를 정적 분석(ESLint). typecheck: TypeScript 타입 오류를 컴파일 전에 탐지(tsc --noEmit). build: 실제 프로덕션 빌드 성공 여부 확인.
예시
코드 작성 후 반드시 pnpm lint && pnpm typecheck && pnpm build 3개 모두 통과해야 커밋.
UUID
범용 고유 식별자
⚙️ 개발 방법론
전 세계적으로 고유함이 보장되는 128비트 식별자. DB 테이블의 기본키(Primary Key)로 자주 사용된다. 숫자 자동증가(1, 2, 3...)보다 보안상 안전하고 분산 시스템에 적합하다.
예시
id UUID PRIMARY KEY DEFAULT gen_random_uuid() — PostgreSQL에서 UUID 자동 생성.
API (Application Programming Interface)
애플리케이션 프로그래밍 인터페이스
⚙️ 개발 방법론
소프트웨어 간의 소통 규칙. 프론트엔드가 백엔드에 "태스크 목록 줘"라고 요청하고 백엔드가 JSON으로 응답하는 방식이 대표적인 REST API다.
예시
GET /api/tasks → 서버가 [{ "id": "...", "title": "UI 구현" }] JSON 반환. TaskFlow는 Server Action으로 별도 API 없이 처리.
SSO (Single Sign-On)
단일 로그인
⚙️ 개발 방법론
하나의 계정으로 여러 서비스에 로그인하는 방식. Google 계정으로 다양한 앱에 로그인하는 "Google로 로그인"이 대표적이다.
예시
Supabase Auth에서 Google OAuth 설정 → "Google로 로그인" 버튼으로 별도 가입 없이 TaskFlow 사용 가능.
Middleware
미들웨어
🛠 프레임워크
요청(Request)이 페이지나 API에 도달하기 전에 실행되는 코드. Next.js에서는 프로젝트 루트의 middleware.ts 파일로 정의한다. 인증 여부 확인, 경로 보호, 리다이렉트 처리 등을 한 곳에서 처리할 수 있다.
예시
미인증 사용자가 /dashboard에 접근하면 미들웨어가 가로채 → /login으로 자동 리다이렉트.
SSR (Server-Side Rendering)
서버 사이드 렌더링
🛠 프레임워크
HTML을 브라우저가 아닌 서버에서 생성해 완성된 페이지를 전달하는 방식. 초기 로딩 속도가 빠르고 SEO에 유리하다. Next.js App Router의 서버 컴포넌트가 기본적으로 SSR 방식으로 동작한다.
예시
태스크 목록 페이지: 서버가 DB에서 데이터를 조회해 완성된 HTML을 반환 → 브라우저는 받은 HTML을 바로 표시.
Hot Reload
핫 리로드
🛠 프레임워크
개발 서버가 실행 중인 상태에서 코드를 수정하면 브라우저를 직접 새로고침하지 않아도 변경 사항이 즉시 반영되는 기능. pnpm dev로 개발 서버를 실행하면 자동으로 활성화된다.
예시
버튼 색상을 blue→red로 수정 저장 → 브라우저 자동 갱신 없이 0.1초 내 변경 반영.
JSONB
제이슨 바이너리
🗄 데이터베이스
PostgreSQL에서 JSON 데이터를 이진(binary) 형식으로 저장하는 타입. 일반 JSON 타입보다 쓰기 속도는 조금 느리지만, 읽기 속도와 인덱싱 성능이 훨씬 뛰어나다. 구조가 유동적인 메타데이터 저장에 주로 사용한다.
예시
metadata JSONB 컬럼에 {"source": "회의록", "page": 3} 저장 → metadata->>'source'로 조회.
코사인 유사도 / Similarity Score
벡터 간 의미적 거리 측정
🗄 데이터베이스
두 벡터 간의 각도를 이용해 유사성을 측정하는 방법. 값은 0~1 범위이며 1에 가까울수록 의미가 비슷하다. pgvector에서는 <=> 연산자로 계산한다. threshold로 최소 점수를 지정해 관련 없는 결과를 걸러낸다.
예시
질문 벡터와 회의록 청크 벡터의 코사인 유사도 0.87 → threshold 0.7 이상이므로 검색 결과에 포함.
Access Token / Refresh Token
액세스 토큰 / 리프레시 토큰
🔒 보안/인증
Access Token: API 호출 시 신원을 증명하는 단기 토큰(수분~수시간 유효). Refresh Token: Access Token이 만료되면 새 Access Token을 발급받기 위한 장기 토큰(수일~수주 유효). Supabase Auth가 로그인 시 두 토큰을 모두 발급한다.
예시
Access Token 만료 → 브라우저가 Refresh Token으로 자동 재발급 → 사용자는 로그아웃 없이 계속 이용.
SQL Injection
SQL 인젝션
🔒 보안/인증
사용자 입력에 악의적인 SQL 코드를 삽입해 DB를 공격하는 방법. 방어책은 파라미터 바인딩(사용자 입력을 SQL 문자열에 직접 연결하지 않는 것)이다. Supabase JS SDK는 내부적으로 파라미터 바인딩을 사용하므로 raw 쿼리 문자열 조합을 피하면 된다.
예시
❌ `WHERE title = '${userInput}'` → ✅ .eq('title', userInput) (Supabase SDK가 자동 바인딩).
HTTPS / SSL·TLS
보안 통신 프로토콜
🔒 보안/인증
브라우저와 서버 간 통신을 암호화하는 프로토콜. HTTPS는 HTTP에 SSL/TLS 암호화를 추가한 것이다. Vercel에 배포하면 자동으로 HTTPS가 적용된다. 로그인·결제 등 민감 정보를 주고받는 모든 서비스에서 필수다.
예시
http://taskflow.app → https://taskflow.app으로 자동 리다이렉트. 주소창 자물쇠 아이콘으로 확인 가능.
GitHub Secrets
깃헙 시크릿
🔒 보안/인증
GitHub Actions에서 API 키 같은 비밀값을 안전하게 저장하고 CI/CD 파이프라인에서 사용하는 기능. 저장 후에는 값을 확인할 수 없으며, 워크플로우 파일에서 ${{ secrets.KEY_NAME }}으로 참조한다.
예시
GitHub 저장소 → Settings → Secrets에 SUPABASE_SERVICE_ROLE_KEY 저장 → CI 빌드 시 자동 주입.
Rate Limiting
요청 속도 제한
🔒 보안/인증
특정 시간 안에 API를 과도하게 호출하지 못하도록 제한하는 정책. 남용·DDoS 공격 방지 목적이다. 제한을 초과하면 서버가 HTTP 429 (Too Many Requests)를 반환한다.
예시
OpenAI API는 분당 60회 호출 제한. 초과 시 429 오류 → 재시도 로직(retry with backoff)으로 대응.
Open Redirect 방지
오픈 리다이렉트 취약점
🔒 보안/인증
로그인 후 ?next=/external-site처럼 외부 URL로 리다이렉트하도록 유도하는 공격. 방어책은 리다이렉트 대상이 같은 도메인의 내부 경로인지 검증하는 것이다. Supabase의 emailRedirectTo도 허용된 URL만 사용해야 한다.
예시
redirectTo 값이 /dashboard처럼 /로 시작하는지 확인 → 외부 URL(http://evil.com)이면 차단.
LLM (Large Language Model)
대규모 언어 모델
🤖 AI/LLM
방대한 텍스트 데이터로 훈련된 AI 언어 모델. 질문 답변, 코드 작성, 번역, 요약 등 다양한 언어 작업을 수행한다. GPT-4, Claude, Gemini 등이 대표적이다.
예시
TaskFlow AI 우선순위 추천: 태스크 목록을 Claude(LLM)에 전달 → "마감이 가장 급한 태스크 3개"를 자연어로 반환.
OpenAI API (gpt-4o-mini · text-embedding-3-small)
오픈에이아이 API
🤖 AI/LLM
OpenAI가 제공하는 AI 모델 API. gpt-4o-mini는 경량 답변 생성 모델(빠르고 저렴). text-embedding-3-small은 텍스트를 1536차원 벡터로 변환하는 임베딩 모델로 RAG에서 문서 검색에 사용한다.
예시
회의록 → text-embedding-3-small로 벡터화 → pgvector 저장 → 질문 벡터와 유사도 비교 → gpt-4o-mini로 최종 답변 생성.
Chunking (청크 분할)
문서 조각내기
🤖 AI/LLM
RAG에서 긴 문서를 LLM이 한 번에 처리할 수 있는 작은 단위(청크)로 쪼개는 작업. 각 청크를 임베딩 벡터로 변환해 저장한다. Overlap은 청크 간 문맥이 끊기지 않도록 앞뒤 청크가 일부 중복되는 영역이다.
예시
10페이지 회의록 → 500자 단위로 분할(overlap 50자) → 각 청크를 벡터화 → DB 저장.
Workflow / 자기 개선 루프
AI 에이전트 작업 흐름
🤖 AI/LLM
AI가 단순 일회성 답변을 넘어 계획 → 실행 → 검토 → 개선을 반복하는 구조. 자기 개선 루프는 AI가 자신의 출력을 평가하고 기준을 통과할 때까지 반복 개선하는 패턴이다. max_iterations로 무한루프를 방지한다.
예시
코드 생성 → 테스트 실행 → 실패 시 오류 분석 → 코드 수정 → 재실행 (최대 3회) → 성공 시 반환.
Protected Route
보호 라우트
🏗 아키텍처
로그인한 사용자만 접근할 수 있도록 인증 여부를 검사하는 페이지 또는 경로. 미인증 접근 시 로그인 페이지로 리다이렉트한다. Next.js에서는 레이아웃 서버 컴포넌트나 미들웨어에서 구현한다.
예시
app/(app)/layout.tsx에서 getUser()→ 미인증 시 redirect('/login') → 이하 모든 페이지 자동 보호.
Webhook
웹훅
🏗 아키텍처
특정 이벤트 발생 시 미리 지정한 URL로 자동으로 HTTP POST 요청을 보내는 방식. 폴링(주기적 확인) 없이 실시간으로 이벤트를 전달할 수 있다. Slack Incoming Webhook이 대표적이다.
예시
RLS 오류 감지 → DB Trigger → pg_net으로 Slack Webhook URL에 POST → Slack 채널에 즉시 알림 발송.
Health Check
헬스 체크
🏗 아키텍처
서버가 정상적으로 동작하는지 주기적으로 확인하는 엔드포인트. 배포 시스템이나 모니터링 도구가 이 URL에 주기적으로 GET 요청을 보내 200 응답이 오는지 확인한다. 문제 발생 시 알림을 발송하거나 자동 재시작을 트리거한다.
예시
GET /api/health → { "status": "ok", "db": "connected" } 반환 → 모니터링 도구가 5분마다 확인.
Observability / Runbook
관찰 가능성 / 런북
🏗 아키텍처
Observability(관찰 가능성): 로그, 지표, 트레이스를 통해 시스템 내부 상태를 외부에서 파악할 수 있는 정도. Runbook: 장애 발생 시 대응 절차를 단계별로 정리한 문서. 온콜 담당자가 사고를 빠르게 해결하기 위해 사용한다.
예시
Runbook: "DB 연결 오류 발생 시 → 1) Supabase Dashboard 확인 → 2) 연결 수 확인 → 3) 재시작 → 4) 슬랙 공유"
Rollback
롤백
🏗 아키텍처
새 배포에서 심각한 버그가 발견됐을 때 이전에 정상 동작하던 버전으로 신속하게 되돌리는 작업. Vercel에서는 대시보드에서 한 번의 클릭으로 이전 배포로 즉시 전환할 수 있다.
예시
v1.2 배포 후 치명적 버그 발견 → Vercel Deployments에서 v1.1 선택 → "Redeploy" 클릭 → 30초 내 롤백 완료.
PDCA (Plan-Do-Check-Act)
계획-실행-확인-개선
⚙️ 개발 방법론
지속적 개선을 위한 4단계 순환 프레임워크. Plan(목표·방법 수립) → Do(실행) → Check(결과 확인·지표 측정) → Act(개선·표준화) → 다시 Plan으로 반복한다.
예시
Plan: "응답시간 200ms 달성" → Do: 코드 최적화 → Check: Vercel Analytics로 측정 → Act: 병목 쿼리 인덱스 추가.
Smoke Test
스모크 테스트
⚙️ 개발 방법론
배포 직후 시스템의 가장 기본적인 기능이 정상 동작하는지 빠르게 확인하는 최소한의 테스트. 전구에 불을 켜보는 것처럼 "연기(smoke)가 나지 않으면 OK"에서 유래한 이름이다.
예시
배포 후 스모크 테스트: 1) 로그인 성공 2) 태스크 목록 로딩 3) 태스크 생성 → 3개 통과 시 배포 완료 선언.
GitHub Actions
깃헙 액션
⚙️ 개발 방법론
GitHub에서 제공하는 CI/CD 자동화 플랫폼. 저장소에 .github/workflows/ 폴더에 YAML 파일을 작성하면, push·PR 등의 이벤트에 자동으로 빌드·테스트·배포 작업을 실행한다.
예시
main 브랜치에 push → GitHub Actions가 pnpm build + typecheck 자동 실행 → 성공 시 Vercel 자동 배포.
Stack Trace
스택 트레이스
⚙️ 개발 방법론
오류 발생 시 어떤 함수에서 어떤 함수로 호출이 이어지다가 오류가 났는지 순서대로 보여주는 정보. 버그의 원인을 찾는 핵심 단서다. 단, 사용자 화면에 노출하면 내부 코드 구조가 공격자에게 노출될 수 있으므로 반드시 서버 로그에만 기록한다.
예시
화면에는 "잠시 후 다시 시도해주세요" 표시 → 서버 console.error에만 스택 트레이스 기록.
CHANGELOG
변경 이력 문서
⚙️ 개발 방법론
소프트웨어 버전별 변경 사항을 날짜와 함께 기록한 문서. 어떤 기능이 추가·수정·삭제됐는지 사용자와 개발팀이 한눈에 파악할 수 있다. Added·Changed·Fixed·Removed 항목으로 분류한다.
예시
## [1.2.0] 2026-06-27
### Added: AI 우선순위 추천 기능
### Fixed: 태스크 소프트 삭제 RLS 오류
